# Team Setup Reference (RBAC)

SpecFlow's RBAC system is git-based and requires no external service. Configuration lives in `.specflow/config.yaml` under the `team` section.

## Solo-Dev Default

When all role lists are empty (the default), RBAC is fully disabled:
- All status transitions are allowed
- Independence checks are skipped
- No CODEOWNERS file is generated

No configuration change needed for solo projects.

## Team Configuration

### Roles

Define roles as named lists of email addresses:

```yaml
team:
  roles:
    reviewer: ["alice@company.com", "bob@company.com"]
    approver: ["carol@company.com"]
    maintainer: ["dave@company.com"]
```

Role names are user-defined. The defaults (`reviewer`, `approver`, `maintainer`) are conventional. Add any roles your organization needs.

### Transition Policies

Map each status transition to the roles that are allowed to perform it:

```yaml
team:
  policy:
    transitions:
      approved: ["approver"]
      implemented: ["maintainer"]
      verified: ["reviewer"]
    verification_statuses: ["verified"]
    directory_ownership:
      _specflow/specs/requirements/: ["approver"]
      _specflow/specs/architecture/: ["approver", "maintainer"]
```

| Field | Description |
|-------|-------------|
| `transitions` | Map of `status: [allowed_roles]`. Empty list = anyone can transition. |
| `verification_statuses` | Statuses that trigger the independence check (default: `["verified"]`) |
| `directory_ownership` | Map of `_specflow/` paths to roles for CODEOWNERS generation |

### Independence Rule

When roles are configured, SpecFlow enforces **implementer != verifier**:

1. When transitioning an artifact to a verification status (e.g., `verified`)
2. SpecFlow checks `git log` for prior authors of that artifact file
3. If the current committer's email appears in the prior authors, the transition is blocked

This prevents self-verification — a key requirement in ASPICE, ISO 26262, DO-178C, and similar standards.

### CODEOWNERS

Running `specflow init` generates a `CODEOWNERS` file from the role configuration. This tells GitHub which team members must review changes to specific directories.

When no explicit `directory_ownership` is configured, all spec directories default to `reviewer` + `approver` roles.

## Enforcement Layers

| Layer | Mechanism | Strength | Setup |
|-------|-----------|----------|-------|
| **Pre-commit hook** | `specflow hook pre-commit` checks RBAC on every commit | Advisory (bypassable with `--no-verify`) | `specflow hook install` |
| **CODEOWNERS** | GitHub requires reviews from designated people | Requires GitHub to enforce | Auto-generated by `specflow init` |
| **Branch protection** | GitHub settings: require PR reviews, require signed commits | Full enforcement — the real gate | Manual setup in GitHub repo settings |

### Recommended GitHub Branch Protection

For full RBAC enforcement, configure in GitHub:

1. **Settings → Branches → Branch protection rules** → `main`
2. Enable "Require a pull request before merging"
3. Enable "Require approvals" (1+ reviewers)
4. Enable "Require signed commits"
5. Enable "Require status checks to pass" → select `specflow-pass-1`

The pre-commit hook catches violations locally (fast feedback). Branch protection catches anything that slips through (hard gate).

## Worked Example: ASPICE Team

```yaml
team:
  roles:
    reviewer: ["qa@company.com"]
    approver: ["architect@company.com", "lead@company.com"]
    maintainer: ["dev1@company.com", "dev2@company.com"]
  policy:
    transitions:
      approved: ["approver"]
      verified: ["reviewer"]
    verification_statuses: ["verified"]
    directory_ownership:
      _specflow/specs/requirements/: ["approver"]
      _specflow/specs/architecture/: ["approver"]
      _specflow/specs/detailed-design/: ["maintainer", "approver"]
      _specflow/specs/unit-tests/: ["maintainer"]
      _specflow/specs/integration-tests/: ["maintainer"]
      _specflow/specs/qualification-tests/: ["reviewer"]
```

With this config:
- Only `architect@` or `lead@` can approve requirements and architecture
- Only `qa@` can mark artifacts as verified
- `qa@` cannot verify their own work (independence rule)
- CODEOWNERS requires appropriate reviewers for each spec directory

## Troubleshooting

| Issue | Cause | Fix |
|-------|-------|-----|
| "author lacks required role" | Your email isn't in the role list for that transition | Add your email to the appropriate role in config.yaml |
| "Independence violation" | You're trying to verify an artifact you implemented | Ask another team member to verify |
| Hook doesn't check RBAC | Roles are empty (solo-dev mode) | Add at least one email to a role list |
| CODEOWNERS not generated | No roles configured | Add team members to roles, then `specflow init` |
