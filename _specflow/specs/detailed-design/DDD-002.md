# Execution Orchestrator

## Purpose

Manages wave-based parallel story execution. Groups stories into dependency-free waves, spawns subagents with isolated context, tracks progress, and handles locks and failures.

## Core Algorithm: Wave Computation

### Function: compute_waves(stories: list[Artifact]) -> list[list[str]]

1. Build dependency graph from story links:
   - If STORY-002 has link target: STORY-001, role: derives_from → STORY-002 depends on STORY-001
   - If two stories share a guided_by link to the same ARCH → they MAY run in parallel (no dependency)
   - If two stories share a specified_by link to the same DDD → they SHOULD run sequentially (likely touch same code)
2. Topological sort the dependency graph
3. Group into waves: stories at the same topological level form a wave
4. Within each wave, stories can execute in parallel

### Data Structures

```python
@dataclass
class ExecutionState:
    current_wave: int
    total_waves: int
    completed: list[str]      # Story IDs
    in_progress: list[str]    # Story IDs
    queued: list[str]         # Story IDs
    failed: list[str]         # Story IDs
```

## Context Isolation Rules

### What each subagent receives:
- The story file (full content)
- Linked DDD artifact (if any, via specified_by)
- Linked ARCH artifact (via guided_by) — interfaces section only
- AGENTS.md project rules
- Relevant source files identified by the story's scope

### What each subagent does NOT receive:
- Other stories or their designs
- Unrelated requirements
- Full _index.yaml
- Other subagent outputs or progress
- Impact-log history

### Context budget per subagent:
- Story: ~500 tokens
- Design doc: ~500 tokens
- AGENTS.md: ~300 tokens
- Source files: ~2000 tokens (estimated)
- Total target: <4000 tokens context, leaving room for generation

## Execution Flow

```
1. Load all stories with status: approved
2. Compute waves
3. For each wave:
   a. Acquire locks for all artifacts touched by wave stories
   b. Spawn subagent per story (parallel)
   c. Wait for all subagents to complete
   d. Collect results
   e. Update artifact statuses (approved → implemented)
   f. Release locks
   g. Auto-commit: git add + git commit with story ID in message
   h. Report wave summary to user
4. If any story fails: move to failed list, continue with remaining waves
5. After all waves: report final summary
```

## Lock Handling

- Before spawning: check .specflow/locks/ for each artifact the story touches
- If locked: move story to next wave's queue
- If lock is stale (PID no longer running): break the lock with warning
- Lock files contain: PID, story ID, timestamp

## Error Handling

- Subagent timeout: configurable, default 10 minutes per story
- Subagent failure: mark story as failed, do not block remaining wave stories
- Lock collision: re-queue, do not fail
- Git commit failure: warn user, continue execution
