# Execution Orchestrator

Manages wave-based parallel story execution via the `specflow go` command. Groups stories into dependency-free waves, spawns subagents with isolated context, tracks progress, and handles locks.

## Implementation Files

- `src/specflow/commands/go.py` — CLI entry point for `specflow go`
- `src/specflow/lib/executor.py` — Subagent execution logic
- `src/specflow/lib/waves.py` — Wave computation and dependency grouping

## Core Algorithm: Wave Computation

### Function: `compute_waves(stories: list[Artifact]) -> list[list[str]]`

Implemented in `lib/waves.py`:

1. Build dependency graph from story links:
   - `depends_on` link → explicit dependency
   - Stories sharing `guided_by` to same ARCH → MAY run in parallel
   - Stories sharing `specified_by` to same DDD → SHOULD run sequentially
2. Topological sort the dependency graph
3. Group into waves: stories at same topological level form a wave
4. Within each wave, stories can execute in parallel

## Context Isolation

### What each subagent receives:
- The story file (full content)
- Linked DDD artifact (via `specified_by`) — interfaces only
- Linked ARCH artifact (via `guided_by`) — interfaces section
- AGENTS.md project rules
- Relevant source files identified by story scope

### What each subagent does NOT receive:
- Other stories or their designs
- Unrelated requirements
- Full `_index.yaml`
- Impact-log history

## Execution Flow (`go.py`)

```
1. Load all stories with status: approved
2. Compute waves via lib/waves.py
3. For each wave:
   a. Acquire locks for all artifacts touched by wave stories
   b. For each story in wave:
      - Spawn subagent with isolated context
      - Subagent implements story, updates code
   c. Wait for all subagents to complete
   d. Collect results
   e. Update artifact statuses (approved → implemented)
   f. Release locks
   g. Report wave summary to user
4. If any story fails: move to failed list, continue remaining waves
5. After all waves: report final summary
```

## Lock Handling (`locks.py`)

- Before spawning: check `.specflow/locks/` for each artifact
- If locked: move story to next wave's queue
- If lock stale (PID no longer running): break lock with warning
- Lock files contain: PID, story ID, timestamp

## Error Handling

- Subagent timeout: configurable, default 10 minutes
- Subagent failure: mark story failed, do not block wave
- Lock collision: re-queue, do not fail
- Git commit failure: warn user, continue execution

## Phase Closure (`done.py`)

The `specflow done` command:
1. Reviews completed work against acceptance criteria
2. Extracts prevention patterns into learned checklists (`.specflow/checklists/learned/`)
3. Updates story status: `implemented` → `verified`
4. Archives the current phase if all stories complete
