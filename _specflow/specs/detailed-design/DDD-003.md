# Review & Lint Pipeline

Assembles review criteria from multiple sources and executes them in two passes: zero-token deterministic checks first, LLM-judged checks second. Implemented across `artifact_lint.py`, `artifact_review.py`, `checklist_run.py`, and supporting lib modules.

## Implementation Files

- `src/specflow/commands/artifact_lint.py` — Zero-token validation engine
- `src/specflow/commands/artifact_review.py` — LLM-judged review with adversarial lenses
- `src/specflow/commands/checklist_run.py` — Checklist execution orchestration
- `src/specflow/lib/lint.py` — Schema, link, status, coverage, quality validators
- `src/specflow/lib/checklists.py` — Checklist assembly and execution
- `src/specflow/lib/analysis.py` — Audit analysis (horizontal, vertical, cross-cutting)
- `src/specflow/lib/techniques/` — Adversarial review techniques

## Artifact Lint (`artifact_lint.py`)

Zero-token validation. The full check set is defined in
`CHECK_NAMES` at `src/specflow/commands/artifact_lint.py:17` — 11 checks
as of v0.2.0:

| Check | Purpose | Implementation |
|-------|---------|----------------|
| `schema` | Validate frontmatter against type schema | `lib/lint.py` schema validation |
| `links` | Verify link targets exist, roles valid | `lib/lint.py` link resolution |
| `status` | Verify status transitions allowed | `lib/lint.py` status hierarchy |
| `ids` | Verify ID uniqueness and format | `lib/lint.py` ID regex |
| `fingerprints` | Detect stale fingerprints | `lib/artifacts.py` recompute + compare |
| `acceptance` | Verify acceptance criteria presence/format on stories & REQs | `lib/lint.py` acceptance parser |
| `conflicts` | Detect contradictory frontmatter or link roles | `lib/lint.py` conflict detector |
| `coverage` | Verify V-model pairs exist | `lib/lint.py` missing pair detection |
| `story-size` | Enforce story size policy (points / SPIDR-slice budget) | `lib/lint.py` size heuristic |
| `chain-report` | Traceability depth distribution | `lib/lint.py` chain depth computation |
| `quality` | EARS, ambiguity, passive voice, compound shall | `lib/lint.py` regex analysis |

### Two Execution Modes
- `--method programmatic` (default): Zero-token, deterministic
- `--method llm`: AI-judged checks for nuanced quality (higher token cost)

## Checklist Assembly (`checklists.py`)

Assembles a unique review criteria set per artifact from four sources:

1. **Artifact-type checklist**: `.specflow/checklists/in-process/{type}-writing.yaml`
2. **Domain-tag checklists**: `.specflow/checklists/shared/*.yaml` — matched by tags
3. **Phase-gate checklists**: `.specflow/checklists/phase-gates/{transition}.yaml`
4. **Learned prevention patterns**: `.specflow/checklists/learned/PREV-*.yaml`

Deduplication: identical check text → keep higher severity.
Sorting: automated items first, then proactive, then reactive.

## Two-Pass Execution

**Pass 1 (Zero-Token):**
- Run all `automated: true` items via Python/shell scripts
- Blocking failure → STOP, do not proceed to Pass 2

**Pass 2 (LLM-Judged):**
- Run all `automated: false` items
- Present artifact + checklist to LLM
- LLM evaluates pass/fail with reasoning

## Adversarial Review Techniques (`techniques/`)

| Technique | Purpose | File |
|-----------|---------|------|
| Devil's Advocate | Challenge assumptions | `devils_advocate.py` |
| Premortem | Imagine failure and work backward | `premortem.py` |
| Assumption Surfacing | Make implicit assumptions explicit | `assumption_surfacing.py` |
| Red/Blue Team | Simulate attack and defense | `red_blue_team.py` |

The `/specflow-artifact-review` skill instructs the agent to select techniques based on artifact type and risk level.

## Quality Checks (`lint.py`)

Regex-based analysis of REQ bodies:

1. **Ambiguity words**: "fast", "user-friendly", "robust", "flexible", "efficient", "approximately", "easy", "simple", "should be able to", "it would be nice if", "etc."
2. **Passive voice**: "shall be <past participle>", "is expected to", "needs to be"
3. **Compound shall**: Multiple "shall" in a single sentence/paragraph
4. **Missing thresholds**: "shall respond quickly" without measurable criterion

All quality findings are warnings (non-blocking) by default.

## Audit Engine Integration (`analysis.py`)

Used by `project-audit.py` for full-project health review:

- **Horizontal analysis**: Per artifact type consistency
- **Vertical analysis**: Per REQ V-model thread coherence
- **Cross-cutting analysis**: Compliance, baseline drift, NFR coverage

Findings with severity `warn` or `error` generate `CHL-*` challenge artifacts (with deduplication to prevent duplicates on re-runs).
