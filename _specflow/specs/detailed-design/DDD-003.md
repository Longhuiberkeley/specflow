# Checklist Assembly Pipeline

## Purpose

Assembles a unique set of review criteria for each artifact by combining four sources: artifact-type checklists, domain-tag checklists, shared checklists, and learned prevention patterns. Ensures review is never generic.

## Assembly Algorithm

### Function: assemble_checklists(artifact: Artifact) -> list[ChecklistItem]

```
Input: artifact with {type, tags, links, status}
Output: deduplicated, prioritized list of checklist items

1. Load artifact-type checklist:
   - Path: .specflow/checklists/in-process/{type}-writing.yaml
   - Example: requirement-writing.yaml for REQ artifacts
   - Always loaded for the artifact's type

2. Load domain-tag checklists:
   - Path: .specflow/checklists/shared/*.yaml
   - For each checklist: read applies_to.tags and applies_to.types
   - If artifact's tags intersect with checklist's applies_to.tags → include
   - If artifact's type matches checklist's applies_to.types → include
   - Example: http-client.yaml loaded for artifacts tagged 'api'

3. Load phase-gate checklists (if phase transition pending):
   - Path: .specflow/checklists/phase-gates/{transition}.yaml
   - Loaded only when the workflow is at a phase boundary
   - Example: planning-to-executing.yaml when moving from planning to executing

4. Load learned prevention patterns:
   - Path: .specflow/checklists/learned/PREV-*.yaml
   - For each pattern: read applies_to.tags
   - If artifact's tags intersect → include
   - These grow over time as the project learns from mistakes

5. Deduplicate: if two checklist items have identical check text, keep the one with higher severity

6. Sort by execution order:
   a. automated: true items first (zero-token)
   b. automated: false, mode: proactive items second
   c. automated: false, mode: reactive items third
```

### Data Structures

```python
@dataclass
class ChecklistItem:
    id: str                    # e.g., CKL-GATE-003-05
    check: str                 # Human-readable check description
    automated: bool            # True = zero-token script, False = LLM-judged
    severity: str              # blocking | warning | info
    mode: str                  # proactive | reactive | standard
    applies_at: list[str]      # [generation, review, decomposition]
    llm_prompt: str | None     # Prompt for LLM-judged items
    source_checklist: str      # Which checklist file this came from

@dataclass 
class ChecklistResult:
    item_id: str
    result: str                # passed | failed | skipped
    detail: str | None         # Explanation for failures
    timestamp: str
```

## Two-Pass Execution

```
Pass 1 (Zero-Token):
  - Run all items where automated: true
  - Use Python/shell scripts for deterministic checks
  - If any blocking item fails → STOP, report, do not proceed to Pass 2

Pass 2 (LLM-Judged):
  - Run all items where automated: false
  - Present the artifact + assembled checklist items to the LLM
  - LLM evaluates each item and returns pass/fail with reasoning
  - Blocking failures must be resolved; warnings are advisory
```

## Logging

Each checklist execution produces a log file:
- Path: .specflow/checklist-log/{timestamp}_{checklist-id}.yaml
- Content: checklist ID, items checked, results, overall pass/fail
- Artifact frontmatter updated: checklists_applied list appended

## Error Handling

- Missing checklist file: warn, skip (do not fail the entire review)
- Malformed checklist YAML: warn, skip that checklist
- No checklists match: warn user that review has no domain-specific criteria
