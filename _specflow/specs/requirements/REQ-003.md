# Trace & Suspect Propagation

The system **shall** detect content changes to artifacts and propagate suspect flags to all downstream-linked artifacts.

## 1. Fingerprint-Based Change Detection

The system **shall** compute and store a SHA256 fingerprint for each artifact:
- Fingerprint **shall** be computed from the normative content (Markdown body after the frontmatter delimiter)
- Fingerprint **shall** be stored in the artifact's `fingerprint` frontmatter field
- On validation, the system **shall** recompute the fingerprint and compare against the stored value
- If the fingerprint mismatches, the system **shall** flag the artifact as modified

## 2. Suspect Flag Propagation

When an artifact's fingerprint changes, the system **shall**:
- Set `suspect: true` on all artifacts that link to the changed artifact (downstream direction)
- For hierarchical artifacts: if a child changes, set the parent's suspect to `partial`; if a parent changes, flag all children
- Create an event record in `.specflow/impact-log/`
- Bump the artifact's `version` number by 1

### Propagation Direction
- If `REQ-001` changes and `ARCH-001` has `links: [{target: REQ-001, role: refined_by}]`, then `ARCH-001` is flagged suspect
- The system traces links in the reverse direction: "who references me?" not "who do I reference?"

## 3. Impact-Log

The system **shall** maintain an append-only impact log in `.specflow/impact-log/`:
- One YAML file per event, named `<ARTIFACT-ID>_<ISO-timestamp>.yaml`
- Each event file **shall** record:
  - `changed`: the artifact ID that changed
  - `change_type`: content_modified, status_changed, created, deleted, split, or merged
  - `fingerprint_old`: previous fingerprint value
  - `fingerprint_new`: new fingerprint value
  - `flagged_suspects`: list of artifact IDs flagged, with the link role that caused the flag
  - `resolved`: boolean indicating whether the suspect has been reviewed

## 4. Typo Cascade Defense

The system **shall** provide three tiers of defense against unnecessary suspect cascades:

### Tier 1: Explicit Intent
- User adds `update_type: minor` to the artifact's frontmatter
- System updates the fingerprint silently, logs the event as minor, removes the field, and skips cascade

### Tier 2: Convenience Command
- `specflow-tweak <file>` recomputes the fingerprint, logs as minor, and skips cascade

### Tier 3: Magnitude Heuristic (Fallback)
- If no `update_type` is set and the change ratio is below 5% of total lines AND only frontmatter changed, auto-classify as minor
- Otherwise: always cascade (conservative default)

## 5. Impact Reporting

The system **shall** provide `specflow-impact` to display:
- All unresolved suspect flags with source artifact lineage
- Change history per artifact from the impact-log
- Oldest unresolved flag age

### Resolving Suspect Flags
- `specflow-impact --resolve <ID>` clears the suspect flag on the specified artifact
- Resolution sets `suspect: false`, updates the impact-log event to `resolved: true`, and records `resolved_by` and `resolved_at`

## 6. Split and Merge Operations

The system **shall** support artifact split and merge:
- **Split**: When an artifact's content diverges into two, present downstream artifacts for reassignment; rewrite links; log split event
- **Merge**: When an artifact is merged into another, rewrite all links pointing to the merged artifact; log merge event

## Acceptance Criteria

1. Given REQ-001 with a downstream link from ARCH-001 via `refined_by`, when REQ-001's body text changes, ARCH-001 is flagged `suspect: true`
2. Given a modified artifact, an impact-log event file is created with artifact-first naming
3. Given a typo fix (1 line changed in a 200-line file), the magnitude heuristic classifies it as minor and skips cascade
4. Given `update_type: minor` in frontmatter, the system updates the fingerprint without propagating suspect flags
5. Given `specflow-impact`, the output shows all unresolved suspect flags with their source artifacts and change types
6. Given `specflow-impact --resolve ARCH-001`, the suspect flag is cleared and the impact-log event is marked resolved
