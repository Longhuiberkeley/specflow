# EARS Quality in Prompts and Lint Safety Net

The system **shall** improve requirements quality at two levels: prompt guidance during creation (shift-left) and deterministic lint checks after creation (safety net).

## Part A: Prompt Guidance (Shift-Left)

The `normative-language.md` reference file **shall** be enhanced with:

1. EARS sentence patterns (`Easy Approach to Requirements Syntax`):
   - **Ubiquitous**: "The \<system\> **shall** \<do something\>."
   - **Event-Driven**: "When \<trigger\>, the \<system\> **shall** \<response\>."
   - **Unwanted Behaviour**: "If \<unwanted condition\>, the \<system\> **shall** \<mitigation\>."
   - **State-Driven**: "While \<state\>, the \<system\> **shall** \<behavior\>."
   - **Optional Feature**: "Where \<feature\> is enabled, the \<system\> **shall** \<behavior\>."

2. Ambiguity word list: words that make requirements untestable (e.g., `fast`, `user-friendly`, `robust`, `flexible`, `efficient`, `approximately`, `easy`, `simple`, `should be able to`, `it would be nice if`, `etc`)

3. Compound `shall` detection: a single requirement containing multiple `shall` clauses **should** be split.

4. Passive voice warning: requirements written in passive voice (`it is expected that`) **should** be reworded to active voice (`the system **shall**`).

The system **shall** update the `/specflow-discover` skill to reference the enhanced guidance so that requirements are authored with proper syntax from the start.

## Part B: Lint Safety Net

A new `--type quality` check **shall** be added to `artifact-lint` that performs zero-token regex analysis on REQ bodies:

1. Detect ambiguity words from a defined word list
2. Detect passive voice patterns (`shall be <past participle>`, `is expected to`, `needs to be`)
3. Detect compound `shall` (multiple `shall` in a single sentence/paragraph)
4. Detect missing quantitative thresholds (`shall respond quickly` without a measurable criterion)
5. Flag each finding as a warning (not blocking)

The quality check **shall** be informational by default but **may** be promoted to blocking via configuration.

## Acceptance Criteria

1. `normative-language.md` includes all 5 EARS sentence patterns with examples
2. `normative-language.md` includes an ambiguity word list with at least 20 entries
3. `normative-language.md` includes guidance on compound shall detection and passive voice avoidance
4. `/specflow-discover` SKILL.md references the enhanced quality guidance
5. `specflow artifact-lint --type quality` performs regex-based quality checks on REQ bodies
6. Quality check detects at least: ambiguity words, passive voice, compound shall, missing thresholds
7. Quality check reports warnings (non-blocking) by default
8. All existing tests continue to pass
