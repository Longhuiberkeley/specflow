# CLI Interface

The system **shall** provide a `specflow` command-line interface as a Python package installed and managed by `uv`.

## Subcommands

### 1. `specflow init`

The system **shall** scaffold a complete SpecFlow project structure:
- Detect the AI platform (`.claude/`, `.gemini/`, `.opencode/`) or prompt the user
- Create the `.specflow/` directory with `config.yaml`, `state.yaml`, and schema files
- Create the `_specflow/` directory with all V-model and work artifact subdirectories
- Append a SpecFlow instructions section to the project's `AGENTS.md` (or chosen file)
- Copy platform-specific skill files into the detected platform's skills directory

### 2. `specflow status`

The system **shall** display a project dashboard:
- Current workflow phase from `state.yaml`
- Artifact counts grouped by type (REQ, ARCH, DDD, UT, IT, QT, STORY, SPIKE, DEC, DEF)
- Artifact status distribution (draft, approved, implemented, verified)
- Link health summary (broken links, orphans, missing verification pairs)
- Suggested next action based on current phase and artifact state

### 3. `specflow validate`

The system **shall** run all validation checks and report results:
- Schema validation: verify YAML frontmatter against artifact type schemas
- Link validation: verify all link targets resolve to existing artifacts
- Status validation: verify status values and transitions are allowed
- ID validation: verify ID uniqueness and format compliance
- Fingerprint validation: detect stale content fingerprints
- Acceptance criteria check: verify every REQ has at least one acceptance criterion

### 4. `specflow-new`

The system **shall** conduct a structured discovery conversation to elicit requirements:
- Phase 1: context-free discovery questions (asked one at a time)
- Phase 2: domain-specific deep-dive based on project classification
- Phase 3: cross-cutting concerns (error handling, security, observability)
- Generate `REQ-*.md` artifacts with proper frontmatter and acceptance criteria

### 5. `specflow-plan`

The system **shall** decompose approved requirements into architecture and stories:
- Read all approved requirements from the artifact registry
- Propose architecture components with interfaces and dependencies
- Generate `ARCH-*` and `DDD-*` artifacts
- Decompose into `STORY-*` artifacts using vertical slicing
- Link all artifacts via appropriate link roles

### 6. `specflow-go`

The system **shall** orchestrate parallel subagent execution per story wave:
- Assign stories to subagents with minimal context
- Track progress and handle filesystem locks
- Auto-commit per task completion

### 7. `specflow-check`

The system **shall** review artifacts against context-specific criteria:
- Assemble criteria from artifact type + domain tags + shared checklists
- Run automated (zero-token) checks first
- Run LLM-judged checks only if automated checks pass

### 8. `specflow-done`

The system **shall** facilitate phase closure:
- Review completed work
- Extract prevention patterns into learned checklists
- Archive the current phase

### 9. `specflow-impact`

The system **shall** report on suspect flag status and change history:
- Show all unresolved suspect flags with source artifact lineage
- Show change history per artifact from the impact-log
- Support resolving individual suspect flags

### 10. `specflow-tweak`

The system **shall** provide a convenience wrapper for minor artifact edits:
- Recompute fingerprint without triggering suspect cascade
- Log the change as `update_type: minor` in the impact-log

### 11. `specflow-baseline`

The system **shall** create and compare immutable project state snapshots:
- `create`: snapshot all artifact statuses and fingerprints
- `diff`: compare two baselines and report differences

## Acceptance Criteria

1. Given a fresh repository, `uv run specflow init` creates all required directories and files without error
2. Given a scaffolded project with no artifacts, `specflow status` reports zero artifacts and suggests running `specflow-new`
3. Given artifacts with invalid frontmatter, `specflow validate` reports the specific missing fields and file paths
4. Given a complete valid project, `specflow validate` returns exit code 0 with all checks passing
5. Given a detected platform (`.claude/`), `specflow init` copies skill files into `.claude/skills/` without prompting
