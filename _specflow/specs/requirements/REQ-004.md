# AI Execution Boundaries

The system **shall** define clear boundaries for what the AI agent does versus what deterministic scripts handle, and shall optimize context window usage through progressive disclosure.

## 1. Skill System

The system **shall** provide modular skill files that extend the AI agent's capabilities:
- Skills **shall** be installed in the detected platform's skills directory (e.g., `.claude/skills/`)
- Each skill **shall** contain a `SKILL.md` file with YAML frontmatter (`name` and `description`)
- The `description` **shall** clearly state when the skill should be triggered
- Skills **shall** follow the progressive disclosure pattern: minimal instructions always loaded, detailed references loaded on demand

### Skill Anatomy
Each skill directory **shall** contain:
- `SKILL.md`: Core workflow instructions (under 500 lines)
- `references/` (optional): Markdown files with detailed constraints, checklists, or schema examples
- `scripts/` (optional): Zero-token deterministic shell or Python scripts

## 2. Deterministic-First Rule

The system **shall** enforce that all deterministic operations are delegated to scripts rather than performed by the AI agent:
- Computing SHA256 fingerprints
- Validating YAML frontmatter against schemas
- Checking link integrity
- Generating sequential IDs
- Rebuilding `_index.yaml` registries
- Filesystem operations (create, move, rename artifacts)

The AI agent **shall** invoke scripts and interpret their output, not perform the computation itself.

## 3. Token Optimization

The system **shall** implement four context optimization strategies:

### Isolate
- Every major task **shall** get a fresh context
- Subagents **shall** receive only the artifacts relevant to their assigned story
- Subagents **shall** return condensed results

### Select
- Hierarchical summarization **shall** be available:
  - Full content (~500 tokens/file)
  - Signatures (~100 tokens) — frontmatter only
  - One-liner (~10 tokens) — title + status
  - Directory summary (~5 tokens) — count + status distribution

### Programmatic-First
- Zero-token tools (grep, AST, lint, git diff, SHA256, TF-IDF) **shall** be invoked before LLM calls
- Validation results from scripts provide structured context for AI judgment

### Compress
- Auto-compaction **shall** occur at 90% context capacity
- Context budget: system 15%, domain 10%, task 40%, tools 20%, history 10%, buffer 5%

## 4. Readiness Assessment

The system **shall** include a readiness assessment mechanism in the discovery skill:
- The AI agent silently evaluates readiness dimensions after each user response
- Dimensions include: problem clarity, user identification, scope boundary, success criteria, technical constraints, data model, non-functional requirements
- Required dimensions must all be satisfied; recommended dimensions require a threshold (e.g., 2 of 3)
- If readiness is met within the first exchange, the system produces lean artifacts (auto-approved)
- If readiness is not met, the system continues discovery with full artifact chain

## 5. Ceremony Adaptivity

The system **shall** adapt ceremony level based on ambiguity, not on which command the user typed:
- Bounded, clear changes produce lean artifacts (minimal metadata, auto-approved)
- Ambiguous, complex changes produce full specification chains
- Lean artifacts grow naturally through the standard workflow — no separate "promote" command

## 6. Phase-Gate Checklists

The system **shall** enforce phase-gate checklists before transitions:
- Six transition checklists: idle→discovering, discovering→specifying, specifying→planning, planning→executing, executing→verifying, verifying→complete
- Each checklist contains items marked as `automated` (zero-token script) or `llm_judged` (token cost)
- Automated items **shall** run first; LLM-judged items run only if automated items pass

## 7. In-Process Checklists

The system **shall** enforce writing constraints during artifact generation:
- Requirement writing: no implementation details, normative language (SHALL/SHOULD/MAY), acceptance criteria required
- Architecture writing: no user-facing behavior, no algorithmic detail, interfaces required
- Design writing: preconditions specified, data structures defined, error handling at boundaries

## Acceptance Criteria

1. Given a skill invocation, the AI agent loads only the SKILL.md initially and reads `references/` files only when instructed by the workflow
2. Given a task requiring fingerprint computation, the AI agent invokes a script rather than computing the hash itself
3. Given a simple, bounded user request ("add a dark mode toggle"), the system produces lean artifacts within 1-2 exchanges
4. Given a complex, ambiguous request ("build a user authentication system"), the system conducts full discovery with Phase 1, 2, and 3 questioning
5. Given a phase transition attempt, the system runs the corresponding gate checklist and blocks the transition if any blocking item fails
