# Provider-Agnostic CI Gate

The system **shall** provide a provider-agnostic CI gate command for server-side RBAC enforcement, with CI adapter templates for popular providers.

## CI Gate Command

A `specflow ci-gate --base <ref> --head <ref>` command **shall**:

1. Read the git diff between two refs (base and head)
2. Extract all artifact status transitions from the diff (files in `_specflow/**.md` with changed `status:` frontmatter)
3. For each transition, run `authorize_status_transition()` from `lib/rbac.py` using the commit author's email
4. For each transition to a verification status, run `check_independence()` from `lib/rbac.py`
5. Report all violations and exit with code 1 if any are found, code 0 otherwise
6. The command uses only git operations -- no provider-specific APIs or environment variables

## CI Adapter Templates

Each CI adapter **shall** be able to generate a ci-gate job that:

- Passes the correct ref variables for its platform (e.g., `github.base_ref` / `github.head_ref` for GitHub, `CI_MERGE_REQUEST_DIFF_BASE_SHA` / `CI_COMMIT_SHA` for GitLab)
- Runs `specflow ci-gate --base <ref> --head <ref>` as a PR check
- Fails the PR build on RBAC violations (independence violations, role violations)

The GitHub Actions adapter **shall** include a `ci-gate` operation in `_OP_JOBS`.

## Acceptance Criteria

1. Given a PR where alice@company.com changed STORY-001 from `implemented` to `verified`, and alice also implemented STORY-001 in a prior commit, `specflow ci-gate --base main --head feature-branch` reports an independence violation and exits with code 1
2. Given a PR with no artifact status changes, `specflow ci-gate` exits with code 0
3. Given a PR where an authorized reviewer changes an artifact to `approved`, `specflow ci-gate` exits with code 0
4. The `specflow ci-gate` command accepts `--base` and `--head` arguments and uses only `git diff` internally
5. The GitHub Actions adapter generates a ci-gate job that runs on pull requests with correct ref variables
6. Adding ci-gate support for a new CI provider requires only a new adapter file -- no changes to `hook.py` or `rbac.py`
