# Artifact Structure

The system **shall** store all specifications and work items as Markdown files with YAML frontmatter in the `_specflow/` directory tree.

## 1. Artifact Format

Every artifact **shall** be a Markdown file with YAML frontmatter and Markdown body.

The frontmatter **shall** include all required fields defined by the artifact's schema.

### Required Frontmatter Fields
- `id`: Unique identifier matching the schema's `id_format` regex
- `title`: Human-readable title of the artifact
- `type`: Artifact type value from the schema's type definition
- `status`: Current lifecycle status (draft, approved, implemented, verified)
- `created`: Creation date in YYYY-MM-DD format

### Optional Frontmatter Fields
- `priority`: Priority level (e.g., high, medium, low)
- `version`: Integer version number, incremented on each semantic change
- `rationale`: Explanation of why this artifact exists
- `tags`: List of string tags for categorization and checklist matching
- `suspect`: Boolean flag indicating downstream impact uncertainty
- `fingerprint`: SHA256 hash of the artifact's normative content
- `links`: Array of link objects with `target` and `role` fields
- `upstream`: Reference to the parent artifact (for hierarchical decomposition)
- `checklists_applied`: Record of checklists executed against this artifact
- `modified`: Date of last modification

## 2. V-Model Specification Artifacts

The system **shall** support six levels of specification artifacts:

| Prefix | Directory | Level | Answers |
|--------|-----------|-------|---------|
| REQ | `_specflow/specs/requirements/` | SWE.1 System | WHAT must the system do? |
| ARCH | `_specflow/specs/architecture/` | SWE.2 Module | HOW is the system structured? |
| DDD | `_specflow/specs/detailed-design/` | SWE.3 Unit | HOW does each part work internally? |
| UT | `_specflow/specs/unit-tests/` | SWE.4 | Verifies DDD |
| IT | `_specflow/specs/integration-tests/` | SWE.5 | Verifies ARCH |
| QT | `_specflow/specs/qualification-tests/` | SWE.6 | Verifies REQ |

### V-Model Pairing Rules
- Each REQ **shall** have a corresponding QT linked via `verified_by`.
- Each ARCH **shall** have a corresponding IT linked via `verified_by`.
- Each DDD **shall** have a corresponding UT linked via `verified_by`.

These pairing rules form the V-model verification chain.

### Level Boundary Rules
- **REQ**: Must not contain implementation details, technology choices, or "how" descriptions
- **ARCH**: Must not describe user-facing behavior (belongs in REQ) or algorithmic detail (belongs in DDD)
- **DDD**: Must not contain system-level concerns or user stories

## 3. Work Artifacts

The system **shall** support four types of work artifacts.

| Prefix | Directory | Purpose |
|--------|-----------|---------|
| STORY | `_specflow/work/stories/` | Units of deliverable user value |
| SPIKE | `_specflow/work/spikes/` | Research and investigation tasks |
| DEC | `_specflow/work/decisions/` | Narrative records and architecture decision records |
| DEF | `_specflow/work/defects/` | Bug tracking with severity and lifecycle |

### Work-to-Spec Linking

Work artifacts **shall** reference spec artifacts via link roles: `implements`, `guided_by`, and `specified_by`.

## 4. Hierarchical Decomposition

The system **shall** support dot-notation decomposition within a level:
- `REQ-001` — top-level requirement
- `REQ-001.1` — first sub-requirement of REQ-001
- `REQ-001.1.1` — maximum depth (3 levels)

### Decomposition Rules
- Maximum depth of 3 levels; anything deeper becomes a separate artifact with `refined_by` link
- Children inherit parent's `type` and `tags` by default (overridable)
- Parent status cannot become `verified` until all children are `verified`

## 5. Artifact Registry (`_index.yaml`)

Each artifact directory **shall** maintain an `_index.yaml` registry with a map of artifact IDs to their title, status, tags, and fingerprint, plus a `next_id` counter for the next sequential ID to assign.

The system **shall** rebuild the registry from filesystem contents if deleted or corrupted — it is a cache, not a source of truth.

## 6. Schema Definitions

The system **shall** define artifact type schemas in `.specflow/schema/`. Each schema defines required fields, optional fields, allowed statuses, allowed link roles, ID prefix, and ID format regex.

Schemas **shall** be additive-only across versions (no field removal, no breaking changes).

## Acceptance Criteria

1. Given a Markdown file with valid YAML frontmatter matching its schema, `specflow validate --type schema` passes
2. Given a REQ artifact containing implementation details, an in-process checklist flags it as a boundary violation
3. Given a hierarchical artifact with 4 dot levels, the system rejects the 4th level and suggests creating a separate artifact with `refined_by`
4. Given a deleted `_index.yaml`, the system rebuilds it from the directory's artifact files
5. Given a new artifact created without updating `_index.yaml`, validation detects the mismatch
