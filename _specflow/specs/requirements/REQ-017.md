# Flexible Artifact Type Extensions

The system **shall** support optional artifact type schemas that users can enable during `specflow init`, extending the 12 built-in types with domain-specific types like hazard, risk, and control — without shipping any industry-standard content.

## Context

Regulated industries (automotive, medical, aerospace, finance) work with artifact types beyond SpecFlow's 12 built-in types (requirement, architecture, detailed-design, unit-test, integration-test, qualification-test, story, spike, decision, defect, audit, challenge). Rather than shipping domain-specific preset packs with standard content — which would require expertise we don't have and risk providing inaccurate compliance guidance — SpecFlow provides the mechanism for users to add custom artifact type schemas and use `/specflow-pack-author` to populate standard content themselves.

The built-in `requirement` type with tags (e.g., `tags: [hazard, safety, asil-d]`) covers most use cases. Optional types are for cases where a genuinely different schema is needed (different status lifecycle, required fields, or allowed link roles).

## Requirements

### 1. Optional Artifact Type Schemas

The system **shall** ship optional artifact type schemas that users can enable:

| Schema | Artifact Type | Prefix | Use Case |
|--------|--------------|--------|----------|
| `hazard.yaml` | hazard | HAZ | Hazard identification and tracking |
| `risk.yaml` | risk | RISK | Risk analysis and assessment |
| `control.yaml` | control | CTRL | Internal controls and audit findings |

Each schema defines: `type`, `prefix`, `id_format`, `required_fields`, `optional_fields`, `allowed_status`, `allowed_link_roles`, `directory`.

### 2. Init Integration

`specflow init` **shall**:
1. Detect optional schemas in `src/specflow/schemas/optional/`
2. Prompt user to select which optional types to enable (or accept none)
3. Copy selected schemas into `.specflow/schema/`
4. Create corresponding directories in `_specflow/specs/`
5. Register selected types in `.specflow/config.yaml`

### 3. Pack-Author Integration

Users create standard content (clauses, checklists) via `/specflow-pack-author`, which generates packs referencing the optional artifact types. No standard content ships with SpecFlow itself.

### 4. Design Principles

- **Built-in types are universal**: The 12 built-in types cover 90% of use cases. Optional types are the exception, not the rule.
- **Tags over types**: When a `requirement` with tags suffices, prefer it over a new type. Only create a new type when the schema genuinely differs.
- **No domain content**: SpecFlow ships schema structure only. Users bring their own licensed standard text via pack-author.

## Acceptance Criteria

1. `specflow init` offers optional artifact types (HAZ, RISK, CTRL) and copies selected schemas
2. After enabling hazard type, `specflow create --type hazard` creates a HAZ artifact
3. Optional types integrate with `specflow artifact-lint` for schema validation
4. Optional types integrate with `specflow trace` for traceability chains
5. Running `specflow init` again does not duplicate already-installed schemas
6. All existing tests continue to pass
