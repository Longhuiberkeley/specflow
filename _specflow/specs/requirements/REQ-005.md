# Cross-Platform & Compliance

The system **shall** support multiple AI CLI platforms and provide compliance-as-code capabilities through baselines and gap analysis.

## 1. Platform Detection

The system **shall** auto-detect the target AI platform during `specflow init`:
- `.claude/` directory present → Claude Code
- `.opencode/` directory present → OpenCode
- `.gemini/` directory present → Gemini CLI
- If none detected → prompt the user to select a platform

## 2. Three-Tier Compatibility

The system **shall** implement three compatibility tiers:

### Tier 1: Universal Layer (AGENTS.md)
- Pure Markdown instructions appended to the project's `AGENTS.md` (or chosen file)
- Works on all AI CLI platforms without modification
- Contains: project uses SpecFlow, where artifacts live, how to check status, brief command reference

### Tier 2: Portable Skill Layer (SKILL.md)
- Standard SKILL.md format with YAML frontmatter
- Progressive disclosure: ~50 tokens at startup, full content on invocation
- Identical across all platforms

### Tier 3: Platform-Specific Adapters
- Platform-specific command formats (Markdown for Claude Code/OpenCode, TOML for Gemini CLI)
- Generated by `specflow init` based on detected platform
- Skills copied into the platform's skills directory

## 3. Ephemeral Local Execution

The system **shall** not require global installation:
- Users install SpecFlow directly into their repository via `uv run specflow init`
- No system-level pollution or global state
- The repository IS the installation target

## 4. Compliance as Code

The system **shall** prove compliance through automated, version-controlled mechanisms:
- **Executable Schemas**: Artifact types defined as YAML schemas, validated deterministically
- **Zero-Token Validation**: Traceability matrices, linkage rules, and phase-gate checklists validated by scripts, not LLM inference
- **Cryptographic Fingerprints**: SHA256 hashes detect unauthorized or untracked changes
- **On-Demand Reporting**: Compliance reports synthesized from git history and impact-log on demand

## 5. Baselines

The system **shall** create and manage immutable project state snapshots:
- Baselines **shall** be stored in `.specflow/baselines/` as YAML files
- Each baseline **shall** include:
  - Name and git tag reference
  - Creation timestamp
  - Snapshot of all artifact statuses and fingerprints
  - Test execution summary (UT/IT/QT pass/fail counts)
- Baselines **shall** be immutable after creation
- `specflow-baseline diff` **shall** compare two baselines and report differences

## 6. Gap Analysis

The system **shall** support compliance gap analysis against imported standards:
- Standards imported as YAML clause files in `.specflow/standards/`
- `specflow-compliance` command **shall** map existing artifacts to standard clauses
- Unmapped clauses reported as compliance gaps
- Mapped clauses reported with linked artifact IDs for audit trail

## 7. Schema Extensibility (Industry Packs)

The system **shall** support industry-specific artifact types via schema packs:
- Base framework ships with V-model core only (REQ, ARCH, DDD, UT, IT, QT, STORY, SPIKE, DEC, DEF)
- Industry packs add new artifact types, directories, and checklists at install time
- Example: `--preset iso26262` adds HAZ (hazard), SG (safety goal), SR (safety requirement) types
- Empty directories for unused artifact types are not created

## Acceptance Criteria

1. Given a project with `.claude/` directory, `specflow init` detects Claude Code and copies skills into `.claude/skills/` without prompting
2. Given a project with no platform directories, `specflow init` prompts the user to select a platform
3. Given a baseline created at v1.0 and another at v2.0, `specflow-baseline diff` reports added, removed, and modified artifacts
4. Given an imported standard with 50 clauses and 45 mapped artifacts, gap analysis reports 5 uncovered clauses
5. Given `specflow init` run on a project with an existing AGENTS.md containing a SpecFlow section, the system prompts about overwriting
