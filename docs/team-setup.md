# Team Setup Guide (RBAC)

SpecFlow's role-based access control is git-based and requires no external service. This guide walks through configuring RBAC for team projects.

## Quick Setup

The fastest way to configure team RBAC is through the guided skill:

```
/specflow-adapter
```

Select **Team Setup** and follow the prompts. The skill writes configuration for you.

## Solo vs Team

| Mode | When | Behavior |
|------|------|----------|
| **Solo** (default) | All role lists are empty | All transitions allowed, no independence checks, no CODEOWNERS |
| **Team** | At least one role has members | Transition policies enforced, independence checks active, CODEOWNERS generated |

Solo projects need zero configuration. Team mode activates automatically when you add team members.

## Configuration

All RBAC configuration lives in `.specflow/config.yaml` under the `team` section.

### Define Roles

Roles are named lists of email addresses. Default roles are `reviewer`, `approver`, and `maintainer`, but you can add any roles your organization needs:

```yaml
team:
  roles:
    reviewer: ["alice@company.com", "bob@company.com"]
    approver: ["carol@company.com"]
    maintainer: ["dave@company.com"]
    safety-manager: ["eve@company.com"]
```

### Define Transition Policies

Map each status transition to the roles allowed to perform it:

```yaml
team:
  policy:
    transitions:
      approved: ["approver"]
      implemented: ["maintainer"]
      verified: ["reviewer"]
    verification_statuses: ["verified"]
```

With this configuration:
- Only `approver` members can set `status: approved`
- Only `maintainer` members can set `status: implemented`
- Only `reviewer` members can set `status: verified`
- Any transition not listed in `transitions` is allowed by anyone

### Independence Rule

When roles are configured, SpecFlow automatically enforces **implementer != verifier**:

1. When transitioning an artifact to a verification status (e.g., `verified`)
2. SpecFlow checks `git log` for prior authors of that artifact file
3. If the current committer's email appears in prior authors, the transition is blocked

This prevents self-verification — a key requirement in ASPICE, ISO 26262, DO-178C, and similar standards.

### Directory Ownership

Control which roles own which spec directories (used for CODEOWNERS generation):

```yaml
team:
  policy:
    directory_ownership:
      _specflow/specs/requirements/: ["approver"]
      _specflow/specs/architecture/: ["approver"]
      _specflow/specs/detailed-design/: ["maintainer", "approver"]
      _specflow/specs/unit-tests/: ["maintainer"]
      _specflow/specs/integration-tests/: ["maintainer"]
      _specflow/specs/qualification-tests/: ["reviewer"]
```

When no explicit `directory_ownership` is configured, all spec directories default to `reviewer` + `approver` roles.

## Enforcement Layers

RBAC enforcement has three layers, each stronger than the last:

| Layer | How | Strength | Setup |
|-------|-----|----------|-------|
| **Pre-commit hook** | `specflow hook pre-commit` checks RBAC on every commit | Advisory (bypassable with `--no-verify`) | Installed by `/specflow-init` |
| **CODEOWNERS** | GitHub requires reviews from designated people | Requires GitHub to enforce PR reviews | Auto-generated by `specflow init` |
| **Branch protection** | GitHub: require PR reviews, require signed commits | Full enforcement — the hard gate | Manual setup in GitHub Settings |

### Recommended GitHub Configuration

For full enforcement, configure branch protection in GitHub:

1. Go to **Settings → Branches → Branch protection rules** → `main`
2. Enable **Require a pull request before merging**
3. Enable **Require approvals** (1+ reviewers)
4. Enable **Require signed commits**
5. Enable **Require status checks to pass** → select `specflow-pass-1`

The pre-commit hook catches violations locally (fast feedback). Branch protection catches anything that slips through (hard gate).

## Worked Example: Automotive ASPICE Team

```yaml
team:
  roles:
    reviewer: ["qa-engineer@auto.com"]
    approver: ["system-architect@auto.com", "project-lead@auto.com"]
    maintainer: ["sw-dev@auto.com", "sw-dev2@auto.com"]
    safety-manager: ["safety@auto.com"]
  policy:
    transitions:
      approved: ["approver"]
      verified: ["reviewer"]
    verification_statuses: ["verified"]
    directory_ownership:
      _specflow/specs/requirements/: ["approver", "safety-manager"]
      _specflow/specs/architecture/: ["approver"]
      _specflow/specs/detailed-design/: ["maintainer", "approver"]
      _specflow/specs/unit-tests/: ["maintainer"]
      _specflow/specs/integration-tests/: ["maintainer"]
      _specflow/specs/qualification-tests/: ["reviewer"]
```

With this configuration:
- System architect or project lead must approve requirements and architecture
- QA engineer is the only one who can mark artifacts as verified
- QA engineer cannot verify their own implementations (independence rule)
- Safety manager is a required reviewer for requirements
- CODEOWNERS ensures GitHub enforces these reviews on PRs

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| "author lacks required role" | Your email isn't in the role list for that transition | Add your email to the appropriate role in `.specflow/config.yaml` |
| "Independence violation" | You're trying to verify an artifact you implemented | Ask another team member in the `reviewer` role to verify |
| Pre-commit hook doesn't check RBAC | All role lists are empty (solo mode) | Add at least one email to a role list |
| CODEOWNERS not generated | No roles configured | Add team members to roles, then run `specflow init` |
| Transitions still allowed despite policy | Role lists might be empty | Check that at least one role has members — this is the on/off switch |
